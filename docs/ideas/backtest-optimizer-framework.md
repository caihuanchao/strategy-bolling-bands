# 通用回测优化框架

## Problem Statement

> 如何在 Web 仪表盘中为任意策略找到最优参数组合，用数据支撑实盘交易决策？

当前从策略信号到实盘交易之间存在断裂：仪表盘只能看信号，无法回答"这组参数历史上到底赚不赚钱"。

## Recommended Direction

**通用优化框架 + Web 仪表盘回测实验室**，分两阶段交付：

### C1 — 核心框架 + 布林带 MVP

构建策略无关的 `ParameterOptimizer` 抽象层，策略通过 `get_optimizable_params()` 自主声明可优化参数。用网格搜索跑布林带 N×M 参数空间，结果缓存到磁盘，在仪表盘新增「回测实验室」标签页展示。

### C2 — 扩展到全部策略 + 智能优化

补齐其余 4 个策略的可优化参数，接入贝叶斯优化处理大参数空间（如 triple_confirm 的 14 个参数），支持跨策略对比。

## 架构概览

```
src/optimizer/
├── __init__.py          # OptimizableParam, OptimizationResult, BaseOptimizer(ABC)
├── grid_search.py       # GridSearchOptimizer — 遍历参数组合
├── bayesian.py          # BayesianOptimizer — C2 阶段，贝叶斯优化
└── cache.py             # 结果缓存（hash key → JSON）

src/backtest.py 改动:
  + run_backtest_with_strategy(df, strategy, params) → BacktestResult

src/strategies/__init__.py 改动:
  + StrategyBase.get_optimizable_params() → List[OptimizableParam]

app.py 新增 API:
  GET  /api/backtest/strategies          → 策略列表 + 可优化参数
  POST /api/backtest/optimize            → 启动异步优化任务
  GET  /api/backtest/optimize/<job_id>   → 轮询结果

templates/dashboard.html 新增:
  + "回测实验室" 标签页（选股 → 选策略 → 设参数范围 → 开始优化 → 结果展示）
```

## 数据流

```
用户选择 symbol + strategy + param_ranges
  → POST /api/backtest/optimize
  → 后台线程启动 GridSearchOptimizer
  → for each (n, m):
      load_data(symbol) → strategy.generate_signals(df, {n, m}) → run_backtest(df)
  → OptimizationResult 写入磁盘缓存
  → 前端轮询 GET .../<job_id> 拿到结果
  → 渲染热力图 + 最优参数卡片 + 净值曲线 + 交易明细
```

### 第二步：选择策略


**交互细节：**

- 如果预计次数过多（> 200），给出黄色提示"参数空间较大，建议缩小范围或增大步长"
- 策略预设按钮：一键填入该策略的推荐搜索范围

### 与现有仪表盘的集成方式

```
现有标签页:  [今日信号] [全部概览] [参数实验室] [回测实验室] ← 新增
                                                      ↑
                                              与参数实验室互补:
                                              - 参数实验室: 预览信号数量变化
                                              - 回测实验室: 验证历史绩效
```

- 股票选择器复用现有 `stocksData`（`/api/stocks` 已返回全量股票列表）
- 策略选择器复用现有 `strategiesList`（`/api/strategies` 已返回策略元数据）
- 数据获取复用现有 `load_data()` 和缓存体系
- 「参数实验室」聚焦"当前快照下信号好不好"，「回测实验室」聚焦"历史上赚不赚钱"，两者互补

## Key Assumptions to Validate

- [ ] **回测引擎兼容所有策略** — 先用布林带验证（已有成功先例），再逐一验证 dual_ma / volume_analysis / triple_confirm / kdj_bollinger_atr
- [ ] **网格搜索性能可接受** — benchmark：布林带 40×20=800 组合在单标的上 < 5 秒
- [ ] **历史最优参数有参考价值** — 明确标注"历史回测不代表未来表现"，不做过度承诺
- [ ] **策略自主声明参数范围合理** — 每个策略提供合理的搜索空间，避免搜出荒谬参数

## MVP Scope (C1)

**包含：**
1. `OptimizableParam` / `OptimizationResult` 数据类
2. `BaseOptimizer` 抽象基类 + `GridSearchOptimizer` 实现
3. `StrategyBase.get_optimizable_params()` 接口，BollingerStrategy 首批实现
4. `run_backtest_with_strategy()` 桥接策略系统与回测引擎
5. 优化结果磁盘缓存（hash key = strategy + symbol + params + date_range）
6. 3 个 API 端点（策略列表、启动优化、轮询结果）
7. 仪表盘「回测实验室」标签页：选股、选策略、设范围、看结果

**不包含：**
- 贝叶斯优化 / 遗传算法
- 布林带以外策略的参数优化
- 市场环境分段优化（D）
- ETF 特殊费用模型（E）
- Walk-forward 验证
- 多标的并行优化

## Extensibility

### → D（市场环境分段优化）
- `BaseOptimizer.optimize()` 预留 `environment_filter` 参数
- `OptimizationResult` 包含 `environment` 字段
- 前端可切换"趋势/震荡/挤压"独立展示每段最优参数

### → E（ETF 定制管线）
- `run_backtest()` 已使用 `config` 中的费率，传入 `cost_override` 即可适配 ETF（免印花税）
- ETF 识别通过 symbol 前缀（sh/ sz/ hk/）

## Not Doing (and Why)

- **不做 Walk-Forward 验证** — 太复杂，C1/C2 阶段先把网格搜索做扎实
- **不做多标的投资组合优化** — 属于另一个问题域，单标的优化先闭环
- **不做参数持久化到 config** — 优化结果作为参考，用户自行决定是否应用
- **不做实盘自动切换参数** — 需要实盘交易接口，超出回测模块范围
- **不做 Jupyter Notebook 方案** — 用户已确认 Web 仪表盘优先

## Open Questions

1. 网格搜索的步长如何定？建议布林带 N 步长 5（5,10,15,...,50），M 步长 0.25（1.0,1.25,...,3.0），共 10×9=90 组合。需要验证 90 次回测在单标的上是否 < 5 秒。
2. 优化任务是同步返回还是异步轮询？建议异步：POST 返回 job_id，前端 2 秒轮询一次。
3. 缓存失效策略？建议按数据日期范围 hash——只要原始 OHLCV 数据不变，相同参数的回测结果永远有效。
